Пока мы обсуждали только индекс типа "ключ - значение", напоминающий поиск по первичному ключу в БД. У какой-то сущности есть специальный идентификатор, на который ссылаются все другие записи, будь то хоть реляционная БД, документно-ориентированная, или будь это вершиной графа, а индекс используется для разрешение подобных ссылок.

Кроме индексов первичного ключа, можно создавать вторичные индексы. в реляционных базах создаются с помощью команды CREATE INDEX, они часто критично важны для выполнения соединений в базе.

Вторичные индексы легко создаются из индекса типа "ключ-значение". Основное отличие в том, что ключи неуникальны, может быть несколько строк с одними и тем же ключом. Эта проблема решается двумя способами: 1. превратив каждое значение в индексе в список соответствующих идентификаторов строк. 2. Сделав все ключи уникальными, добавив к ним идентификатор строки. В любом случае b дерево и LSM деревья можно использоваться как вторичные индексы.
